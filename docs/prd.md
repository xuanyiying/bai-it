# 掰it — 产品需求文档

## 用户痛点

英文阅读中最大的障碍不是单词，而是长句——多层嵌套的从句、倒装、插入语让人一读就晕。现有工具要么只做翻译（失去学习机会），要么只做词典（解决不了句子结构问题）。

更具体地说，用户在两个场景下有不同的痛点：

1. **刷信息流（推特、Reddit）时**：需要快速筛选哪些内容值得细看，但英文长句让扫读速度大打折扣。很多句子语法不复杂，但信息密度高、并列结构多，一眼扫不过去。这不是学习问题，是效率问题。
2. **决定细读一篇文章时**：面对整页英文会被"吓退"。需要适度的辅助把难度降到可接受的范围（i+1 ~ i+3），同时保留足够的挑战让阅读过程产生真正的学习效果。

## 设计原则

1. **装完即用**：用户装插件就能用，不需要 API key、不需要注册。基础功能（拆句 + 生词释义）零配置
2. **数据归用户**：所有数据存浏览器本地，不上传、不泄露
3. **学习而非翻译**：帮用户理解句子结构，不是替用户读
4. **不制造依赖**：工具的目标是让用户逐渐不需要它。扫读提供效率，细读提供成长，两者缺一不可
5. **渐进增强**：基础版本地完成，配了 API key 后体验升级。两层之间平滑过渡，不是两个割裂的产品

## 三层体验模型

掰it 的体验分三层，每层在上一层基础上增强。核心原则：**装了就能用，点了更好用，配了 key 最好用**。

### 第一层：装完即用（无需任何配置）

- **自动扫读**：所有英文网页自动启用，本地规则引擎拆分长句 + 标注生词，零延迟、零 API 成本
- **本地生词释义**：离线词典 + 行业术语包，hover 虚线显示中文释义。**这是底线体验——拆分后的句子中，生词必须有释义**
- **站点开关**：用户可通过 Popup 关闭特定网站，设置持久化（chrome.storage.sync）
- 管理端展示示例数据 + 引导提示

### 第二层：手动触发（无需 API key）

- **手动掰句**：用户对任何句子点击触发按钮 → 本地强制拆分（忽略长度阈值，尽力拆）
- 适用场景：自动扫读没拆的短句，用户觉得看不懂，主动点一下

### 第三层：AI 深度分析（需配置 API key）

- **手动触发升级**：有 API key 时，手动触发改为发 AI，返回句式分类、句式讲解、表达技巧、语境化生词释义
- **管理端全功能解锁**：总览、难句集、每日回味、设置四个 Tab 全部可用
- **学习数据积累**：手动触发的句子自动存入 IndexedDB 待分析队列，管理端按需处理（见「数据采集策略」）

## 核心概念：自动扫读 + 手动掰句

不再区分"扫读模式"和"细读模式"。所有英文网页统一一套逻辑：**自动扫读处理全场，手动触发按需深入**。

### 自动扫读（被动，所有英文网站）

- **目标**：快速理解，不卡住
- **触发**：所有英文网页自动启用（用户可通过 Popup 关闭特定站点）
- **不再按站点判断模式**：删除原来的 `detectReadingMode` 站点列表逻辑
- **分块策略**：本地规则即时拆分
  - 按长度拆：超过阈值的句子自动拆（阈值可调：短/中/长三档，默认中）
  - 在逻辑转换点断行：并列、转折、条件、因果、补充、从句引导等
  - 零延迟、零 API 成本
- **排版**：保留缩进层级，体现句子内部结构
- **生词**：hover 虚线提示
  - 离线词典 + 行业术语包标注（V1 内置 AI 行业包）
  - 用户标记"已掌握"的词自动跳过
  - **拆分后的句子中，生词释义必须正常显示**（底线体验）

### 手动掰句（主动，用户点哪句拆哪句）

- **目标**：用户觉得某句话看不懂，主动"掰"一下
- **触发**：每个段落/句子旁显示手动触发按钮，用户点击触发
- **无 API key**：本地强制拆分（忽略长度阈值，尽力拆）
- **有 API key**：发 AI 做深度分析（句式分类、讲解、表达、语境化释义）
- **数据采集**：手动触发的句子存入 IndexedDB 待分析队列（见「数据采集策略」）

### 站点控制

- **默认全开**：所有英文网页自动启用扫读
- **站点级关闭**：Popup 中 toggle 关闭某个域名，持久化到 chrome.storage.sync，重启浏览器保留
- **页面级暂停**：Popup 中大按钮可暂停/恢复当前页面的处理

### Popup 设计

Popup 是用户日常使用的快捷控制面板，只放高频操作，不放 AI 配置等低频设置。

**控制区（顶部）：**

- **大按钮**：当前页面的一键开关。两种状态切换——"显示原文"（当前正在拆分，点击可关闭）/ "拆分显示"（当前未拆分，点击可开启）
- **站点级开关**：一行文字"总是拆分 x.com" + toggle 开关。控制整个域名是否启用，关闭后大按钮变灰不可用。默认开启

**拆分设置区：**

- **辅助力度**（一个滑杆，1-5 档）：控制自动扫读拆多少句子。底层控制拆分粒度（chunkGranularity）和扫读最小词数阈值（scanThreshold），用户只看到一个"少拆 ↔ 多拆"的滑杆。每档配一句话说明（如"适中 — 复杂句自动拆，简单句不打扰"）
- **显示方式**（分段选择器 + 预览区）：控制拆完后怎么显示。三个选项：
  - **详细**（对应 L5）：分行 + 缩进，一眼看清主从关系
  - **简洁**（对应 L3）：只分行，不加额外标记
  - **轻微**（对应 L1）：不分行，次要部分变淡
  - 选项名称用干预程度命名（详细/简洁/轻微），下方显示一句话说明 + 实时预览效果

**底部：**
- "更多设置 →" 链接，跳转到 Options 页的设置 Tab

**原型文件**：`_local/mockups/mockup-popup.html`（可在浏览器中打开交互体验）

## 句式分类体系

AI 每次处理复杂句子时，同时返回该句子的句式分类。分类从以下固定列表中选取（AI 返回英文 key，前端映射为用户友好的名称 + 一句话解释）：

| key | 用户看到的名称 | 一句话解释 |
|-----|--------------|-----------|
| `insertion` | 插入补充 | 句子中间插了一段来补充说明 |
| `background_first` | 先说背景 | 先交代条件/原因/时间，再说正事 |
| `nested` | 层层嵌套 | 句子里套句子，要剥好几层 |
| `long_list` | 一口气列举 | 连着列了好几样东西 |
| `inverted` | 倒着说 | 正常语序被颠倒了 |
| `long_subject` | 超长主语 | 主语说了一大串才到动词 |
| `omission` | 省略跳跃 | 省掉了一些词，要自己补上 |
| `contrast` | 对比转折 | 前后两半在对比或转折 |
| `condition` | 条件假设 | 如果……就……，但说法不直白 |
| `long_modifier` | 超长修饰 | 用一大段话来描述一个东西 |

设计原则：
- 名称用通俗但稳定的措辞，用户能看懂，同时能作为筛选标签归类
- AI 在 prompt 中被约束只能从以上列表选取，保证标签收敛
- 前端映射表与 prompt 解耦，改措辞不需要改 prompt
- 兜底：万一 AI 返回列表外的值，显示为"其他"

## 功能范围

### P0：基础体验（零配置可用）

- 所有英文网页自动扫读（本地规则拆分 + 生词标注），不依赖站点列表
- 本地规则引擎拆分（零延迟、零 API 成本）
- **本地生词释义**：离线词典 + 行业术语包，拆分后的句子中生词必须有中文释义显示
- 分块结果渲染（保留缩进层级）
- 手动触发按钮：无 API key 时本地强制拆分（忽略阈值）
- 主流站点专用 DOM 选择器（X、Reddit、Medium 等）+ 通用选择器兜底

### P0.5：AI 增强（需配置 API key）

- 插件直调 AI API（Gemini 格式 + OpenAI 兼容格式）
- 缓存机制（同一句子不重复调用 API）
- 手动触发升级：有 API key 时，手动触发改为发 AI，返回分块结果、句式分类、句式讲解、学会表达、生词释义、学习价值标记
- 手动触发的句子自动存入 IndexedDB 待分析队列

### P1：学习系统

#### 数据采集策略

采集和分析解耦：**浏览时存原始数据，管理端按需发 AI 分析**。

**浏览时采集（零成本）：**
- **自动扫读采集**：扫读过程中遇到的句子 + 生词 + 来源 URL + 时间戳，存入 IndexedDB `pending_sentences` 表
- **手动触发采集**：用户手动掰的句子优先级更高，标记 `manual: true`
- **生词遭遇记录**：每次生词出现自动记录一条遭遇（词、上下文句子、来源 URL、时间），用于频率统计

**管理端按需分析（懒处理）：**
- 用户打开管理端（难句集 Tab）→ 从 `pending_sentences` 拉取未分析的句子
- **按页处理**：每页显示 10 条，只对当前页的 10 条发 AI 分析
- **逐条填充**：先显示原始句子骨架，AI 返回一条就渲染一条（不是白屏等）
- **翻页再发**：用户真的翻到下一页时才处理下一批 10 条
- **结果缓存**：分析过的句子存入 `learning_records` 表，下次打开不重复分析
- **无 API key 时**：管理端展示示例数据 + 引导配 key，不触发 AI

**为什么不预处理：**
- 用户一天可能扫读几百上千条推文，全量发 AI 浪费且不必要
- 用户主动打开管理端、主动翻页 = 真的想学的，比任何自动筛选都精准
- 成本与用户实际学习行为成正比，而非浏览量成正比

#### 难句卡片内容

每条 AI 处理过的句子存储为一张难句卡片，包含以下层级：

1. **原句**（收起状态可见）
2. **句式标签**（收起状态可见，如"插入补充"，hover 显示一句话解释）
3. **句子分块**（展开后可见，结构拆解）
4. **句式讲解**（用大白话解释这句话为什么难读）
5. **学会表达**（从句中提取一个可套用的表达模板 + 简化仿写示例，目标是让用户学会用类似结构回复/写作）
6. **生词**（词 + 语境释义 + 标记掌握按钮）
7. **来源 URL**

#### 管理界面（插件 Options 页，React）

整体视觉风格：讲究的字体组合（有辨识度的标题字体 + 清晰的正文字体）、克制的配色（3-5 色）、交错入场动画（自定义缓动曲线）、抽象 CSS 装饰（渐变圆、几何线条，不用插图）。

**管理端三种状态：**

四个 Tab（总览、每日回味、难句集、设置）**始终可见**，不做单独的引导页。区别在于数据内容和顶部提示条：

**① 未配置 API key：**
- 四个 Tab 展示**精心设计的示例数据**（硬编码，模拟"用了 3 天"的效果），正常视觉，不灰度
- 内容区顶部显示提示条：「以下是示例数据。配置 API Key 后，你浏览英文网页时就会自动积累真实的学习数据。[去设置 →]」
- 设置 Tab 始终可操作，用户可直接配置 API key

**② 已配置 API key 但无分析数据：**
- 四个 Tab 继续展示示例数据
- 提示条文案变为：「以下是示例数据。去浏览几篇英文网页，对感兴趣的句子点掰一下，你的学习数据就会出现在这里。」
- 难句集 Tab：如果 `pending_sentences` 有待分析句子，显示待分析列表（打开时按需发 AI）

**③ 有真实数据（learning_records ≥ 1 条）：**
- 所有 Tab 同时切换到真实数据，示例数据和提示条一起消失

提示条在设置 Tab 下不显示（设置页本身就是操作界面）。

**四个 Tab：**

**1. Dashboard**
- 统计卡片（遇到的难句数、生词数、已掌握词数、今日新增）
- 最近的难句（每条显示句式标签 + 截断原文）
- 高频生词（按遭遇次数排序）
- 本周回顾（合并在 Dashboard 内，不单独成页）：AI 生成的周总结 + 常见句式分布 + 词汇精选。按需生成——用户打开 Dashboard 时触发，生成后缓存，同一周内不重复生成

**2. 每日回味**
- 设计理念：**回味，不是复习。** 不制造队列、不打卡、不评分、不记连续天数。用户打开看一眼，10 秒搞定，没有"你还没做完"的压力
- 一屏完事，不翻页

- **断句练习（核心）**
  - 展示一个裸句（无任何标记的原始英文长句）
  - 用户凭语感点击句子中觉得该断开的位置，加分隔标记
  - 点"看答案"，显示 AI 当初的分块结果，用户自行对比
  - **不打分、不判对错。** 用户自己感受"我断对了"还是"原来应该在这里断"
  - 训练的是断句直觉，不是记忆某个句子怎么断

- **选句逻辑**
  - AI 分块时同时返回一个"学习价值"布尔标记（有嵌套结构、典型句式 → 值得练；纯并列堆砌、太短 → 不值得）
  - 从当天标记为"值得练"的句子中随机挑一个
  - 当天没掰过句子 → 从历史中捞一个没练过的
  - 每次出新句子，不重复旧句子（断句练的是感觉不是记忆，新句子比重复旧句子更有价值）

- **高频词汇（附带）**
  - 从当天所有掰过的内容中独立挑选 3-5 个高频词，不跟句子绑定
  - 可标记"已掌握"，下次自动跳过
  - 按遭遇频次排序，优先展示反复遇到但还没标记掌握的词

- **轻量数字**
  - "你这周掰了 X 句"——成就感，不是压力

- **明确不做的事**：进度条、对错评分、连续天数/打卡、队列（"还剩 N 个未复习"）、间隔重复算法

**3. 难句集**
- 所有 AI 处理过的复杂句子，按时间倒序排列
- 收起状态：原句（截断）+ 句式标签 + 来源域名 + 时间 + 生词数
- 展开状态：完整的难句卡片内容（分块、讲解、表达、生词）
- 筛选：按句式标签筛选（如只看"层层嵌套"类的句子）
- 搜索：按关键词搜索原句或生词

**4. 设置**
- AI 提供商和模型选择
- API key 配置（加密存储）
- 分块灵敏度调整
- 其他偏好
