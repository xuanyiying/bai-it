# 掰it — 测试与验收文档

## 测试理念

**真正的验收标准是用户体验**：打开推特能不能拆、拆得对不对、快不快。单元测试保证核心逻辑不出低级 bug，浏览器验收测试保证东西真的能跑。两者缺一不可。

开发节奏：**做一步、测一步**。不攒到最后再测。

## 测试分类

| 类型 | 工具 | 谁跑 | 验什么 |
|------|------|------|--------|
| 单元测试 | Vitest | Claude 自动 | 纯逻辑：分块规则、复杂度评分、请求格式、数据层 |
| 浏览器验收 | Chrome CDP 脚本 | Claude 自动 | 真实效果：扩展加载后页面行为、DOM 变化、性能 |
| 人工确认 | 用户操作 | 用户 | 视觉效果、交互体感、主观质量判断 |

## 测试基础设施

### 单元测试（Vitest）

```bash
npm test              # 跑全部
npm test -- scan-rules # 跑指定文件
```

测试文件放在 `src/__tests__/` 下，文件名对应被测模块（如 `scan-rules.test.ts`）。

### 浏览器验收测试（Chrome CDP）

**调试用 Chrome 配置**

独立配置目录 `~/.chrome-debug-profile/`，X/Twitter 已登录，与用户主力 Chrome 互不影响。

启动命令：
```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.chrome-debug-profile" \
  &>/dev/null &
```

验证连接：`curl -s http://localhost:9222/json/version`

> Chrome 新版本要求 `--remote-debugging-port` 必须搭配非默认 `--user-data-dir`，否则端口不生效。

**加载扩展进行测试**

构建后用以下命令启动（同时加载扩展）：
```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.chrome-debug-profile" \
  --load-extension=./dist \
  &>/dev/null &
```

**验收脚本**

验收脚本放在 `tests/acceptance/` 下，用 Python + websockets 通过 CDP 连接浏览器，执行 JS 检查 DOM 和性能。每个脚本可独立运行，输出 PASS/FAIL。

```bash
python3 tests/acceptance/scan-mode.py   # 扫读模式验收
python3 tests/acceptance/deep-read.py   # 细读模式验收
```

| 站点关闭记忆 | 浏览器验收 | Popup 关闭某站点 → 重启浏览器 → 该站点仍为关闭状态 |
| 性能无退化 | 浏览器验收 | X 首页扫读处理耗时与重构前持平（±20%） |

#### 阶段二：数据采集 + 管理端懒处理

| 验收项 | 类型 | 通过标准 |
|--------|------|----------|
| pending_sentences 表 | 单元测试 | CRUD 操作正常（fake-indexeddb），字段包含 text/source_url/manual/analyzed/created_at |
| schema 升级兼容 | 单元测试 | 从旧版本升级后，已有表数据不丢失，新表正确创建 |
| 扫读自动采集 | 浏览器验收 | 浏览英文网页 → pending_sentences 表中出现采集的句子（DevTools 查 IndexedDB） |
| 手动触发标记 manual | 浏览器验收 | 手动掰的句子在 pending_sentences 中 `manual: true` |
| 不重复采集 | 单元测试 | 同一句子（text 相同）不重复插入 |
| 难句集-按需加载 | 浏览器验收 | 打开管理端难句集 Tab → 显示待分析句子列表 → 自动对当前页（10 条）发 LLM 分析 |
| 难句集-逐条填充 | 浏览器验收 | 分析过程中先显示原始句子骨架，LLM 返回一条渲染一条，不白屏等待 |
| 难句集-翻页触发 | 浏览器验收 | 翻到下一页 → 才对下一批 10 条发 LLM，不预加载 |
| 难句集-结果缓存 | 浏览器验收 | 分析过的句子存入 learning_records → 关闭管理端再打开 → 已分析的直接显示，不重复发 LLM |
| 难句集-无 API key | 浏览器验收 | 未配置 API key → 难句集展示示例数据 + 提示条，不触发 LLM |
| 采集量合理 | 人工确认 | 刷 X 十分钟后查 pending_sentences 条数，确认不会无限膨胀（应有去重） |

## 测试数据

### 英文句子测试集

在 `tests/fixtures/sentences.json` 中维护一组固定的英文句子（从真实推特/Reddit 抓取），用于：
- scan-rules 单元测试的输入
- rule-engine 单元测试的输入
- 浏览器验收测试的对照参考

句子集应覆盖：
- 短句（< 10 词，不应被拆）
- 中等并列句（and/or/but 连接）
- 转折/条件/因果句
- 多层嵌套从句
- 信息密度高但语法简单的长句（不应被过度拆分）
- 中文推文（不应被处理）

- `gemini-valid.json` — Gemini 格式正常响应
